# Research: Optimal Agentic Web Development on Next.js

**Date**: 2026-02-22T16:00:00-05:00

## Research Question

Given that Next.js App Router + shadcn/ui + Tailwind CSS has been selected as the frontend stack (see [[2026-02-22-saas-dashboard-framework-for-agent-development|framework selection research]]), how do you optimally set up a Next.js project for AI agent-driven development? Covers: agent guardrails, AGENTS.md structure, shadcn/ui MCP workflow, testing strategy, project structure, DX toolchain, starter templates, auth/state/data patterns, and deployment.

## Summary

Vercel has built first-party infrastructure for agent-driven Next.js development. The key discovery: **AGENTS.md with bundled docs achieves 100% pass rate on Vercel's agent evals** — outperforming all skills-based approaches (53–79%). The optimal setup is a layered system: (1) `AGENTS.md` generated by `npx @next/codemod@canary agents-md` pointing to `node_modules/next/dist/docs/`, (2) shadcn/ui MCP server for component registry access, (3) Next.js MCP server for runtime diagnostics, (4) Playwright + Chrome DevTools MCP for visual verification, (5) ESLint with `eslint-config-next/core-web-vitals` + strict TypeScript for fast feedback, and (6) a hybrid colocation project structure with private `_components/` folders.

### Next.js MCP Server (Runtime Diagnostics)

### The 9 Canonical Agent Failure Modes

**Source**: [vercel.com/blog/common-mistakes-with-the-next-js-app-router-and-how-to-fix-them](https://vercel.com/blog/common-mistakes-with-the-next-js-app-router-and-how-to-fix-them)

| # | Mistake | Fix |
|---|---------|-----|
| 1 | Fetching from Route Handler inside Server Component | Call data function directly |
| 2 | Expecting Route Handlers to be dynamic by default | Export `dynamic` config or use dynamic fetch |
| 3 | Using Route Handlers for mutations from Client Components | Use Server Actions |
| 4 | Suspense boundary inside async component | Place Suspense ABOVE/OUTSIDE the async component |
| 5 | `useSearchParams` when Server Component props suffice | Use `searchParams` page prop |
| 6 | Context providers not isolated as Client Components | Wrap in `'use client'` component |
| 7 | Adding `'use client'` to every component | Only add at interactive leaf components |
| 8 | Forgetting `revalidatePath()` after Server Action mutations | Always revalidate |
| 9 | `redirect()` inside try/catch | Call redirect AFTER try/catch |

### Next.js 15+ Breaking Changes (Not in Training Data)

**Async params** (hard error in Next.js 16):
```tsx
// WRONG (pre-15 pattern)
export default function Page({ params }: { params: { slug: string } }) {
  const { slug } = params;  // HARD ERROR in Next.js 16

// RIGHT
export default async function Page(props: { params: Promise<{ slug: string }> }) {
  const { slug } = await props.params;
```

**All async APIs** now require await: `cookies()`, `headers()`, `draftMode()`, `params`, `searchParams`.

**Fetch caching** changed: not cached by default in Next.js 15+. Use `cache: 'force-cache'` or `'use cache'` directive (Next.js 16).

**Next.js 16 specific**: `proxy.ts` replaces `middleware.ts`, `cacheComponents: true` replaces `experimental.ppr`, React Compiler is stable (`reactCompiler: true`).

### Recommended CLAUDE.md Structure

```md
@AGENTS.md

## Next.js App Router — Project Rules

### Stack
- Next.js 16 App Router (NOT Pages Router)
- shadcn/ui + Tailwind CSS v4 + Radix UI
- TypeScript strict mode
- Drizzle ORM + PostgreSQL

### Server vs Client Components
- Default: Server Components. Add 'use client' only for: state, event handlers, browser APIs
- DO NOT mark layouts, pages, or data-fetching components as 'use client'
- 'use client' propagates to all imports — apply at leaf level only

### Data Fetching
- Server Components: fetch data directly with async/await
- Mutations: use Server Actions ('use server'), NOT Route Handlers
- After mutations: always call revalidatePath() or revalidateTag()
- Route Handlers: ONLY for webhooks, external API consumers, custom response formats

### Async APIs (BREAKING — Next.js 15+)
- params and searchParams are Promises — ALWAYS await them
- cookies(), headers(), draftMode() — ALWAYS await

### Prohibited Patterns
- getServerSideProps / getStaticProps (Pages Router only)
- Synchronous params access
- redirect() inside try/catch
- Suspense inside async data-fetching components
- useEffect for data fetchable in a Server Component
- Barrel exports (index.ts) in application code

### Component Installation
- shadcn primitives: `npx shadcn@latest add -y <component>` — NEVER hand-write
- Check shadcn MCP for APIs before generating custom implementations
```

### Why This Structure

- **Private `_components/` and `_lib/` folders** colocate route-specific code exactly where agents look. No separate `features/` directory to discover.
- **Route groups `(auth)`, `(dashboard)`** separate layouts without URL impact.
- **`components/ui/`** is CLI-managed shadcn — agents should NEVER hand-write these.
- **No barrel exports** — `import { Button } from '@/components/ui/button'` is unambiguous. Barrel exports cause 68% more module loading and break `optimizePackageImports`.

### Component Placement Rules

1. Used by exactly one route → `route/_components/`
2. Used by 2+ routes in same section → section layout `_components/`
3. Used across sections → `components/` (global)
4. shadcn primitive → `components/ui/` (CLI-managed)

### Naming Conventions

| File type | Convention | Example |
|---|---|---|
| Route files | lowercase kebab-case | `user-settings/page.tsx` |
| Components | PascalCase | `UserAvatar.tsx` |
| Hooks | camelCase with `use` prefix | `useCurrentUser.ts` |
| Actions | kebab + `.actions.ts` | `user.actions.ts` |
| Data | kebab + `.data.ts` | `user.data.ts` |
| Types | kebab + `.types.ts` | `user.types.ts` |

### Monorepo Package Imports

```json
// packages/ui/package.json
{
  "name": "@workspace/ui",
  "exports": {
    "./components/*": "./src/components/*.tsx",
    "./hooks/*": "./src/hooks/*.ts",
    "./lib/*": "./src/lib/*.ts",

```

### TypeScript Configuration

```jsonc
{
  "compilerOptions": {
    "strict": true,
    // Beyond strict — agent error detection:
    "noUncheckedIndexedAccess": true,       // arr[0] is T | undefined
    "exactOptionalPropertyTypes": true,     // absent ≠ undefined
    "noPropertyAccessFromIndexSignature": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true,
    "noImplicitOverride": true
  }
}
```

```typescript
// next.config.ts — typed routes + env
const nextConfig: NextConfig = {
  typedRoutes: true,             // <Link href="/typo"> = type error
  experimental: { typedEnv: true },
}
```

**`noUncheckedIndexedAccess`** is the single highest-value flag — agents frequently write `arr[0].property` without null checks.

### Linting: ESLint (Not Biome) for Agents

**Critical**: `next lint` was removed in Next.js 16. Run ESLint directly.

```javascript
// eslint.config.mjs
import { defineConfig, globalIgnores } from 'eslint/config'
import nextVitals from 'eslint-config-next/core-web-vitals'
import nextTs from 'eslint-config-next/typescript'

export default defineConfig([
  ...nextVitals,
  ...nextTs,
  globalIgnores(['.next/**', 'out/**', 'build/**', 'next-env.d.ts']),
])
```

**Why ESLint over Biome**: Biome is 10-25x faster but **missing `@next/eslint-plugin-next` rules** — specifically `no-async-client-component`, `no-img-element`, `no-html-link-for-pages`. These catch exactly the mistakes agents make. Use ESLint for correctness, optionally add Biome for formatting speed.

### PostToolUse Hooks (Frontend Equivalent)

Three-layer hook stack (analogous to backend ruff/ty/import-linter):

| Layer | Tool | Speed | When |
|---|---|---|---|
| 1 | `biome check --apply` | < 500ms | Every file write |
| 2 | `eslint --fix <file>` | 1–5s | Every file write |
| 3 | `tsc --noEmit` (incremental) | 5–30s | Pre-commit or async |

**Future**: TypeScript 7.0 (Go-native `tsc`) will deliver 10x speedup — 78s → 7.5s on VS Code-scale projects.

### Formatting

Use **Biome** for formatting (single `biome.json`, 25x faster than Prettier). Known gap: Tailwind CSS v4 class sorting not yet supported.

## 5. Testing Strategy

### Stack

| Layer | Tool | What it tests |
|---|---|---|
| Unit | Vitest + Testing Library | Synchronous Client Components |
| E2E | Playwright | Async Server Components, full flows |
| Visual | Chromatic + Storybook | Component visual regression |
| Agent verification | Playwright MCP + Chrome DevTools MCP | Live UI inspection |

**Critical constraint**: Async Server Components **cannot be unit tested** by any framework (Vitest, Jest, Cypress). E2E via Playwright is the only path.

### Vitest Setup

```bash
pnpm add -D vitest @vitejs/plugin-react jsdom @testing-library/react @testing-library/dom vite-tsconfig-paths
```

```typescript

import react from '@vitejs/plugin-react'
import tsconfigPaths from 'vite-tsconfig-paths'

export default defineConfig({
  plugins: [tsconfigPaths(), react()],
  test: { environment: 'jsdom' },

```

```json
// .mcp.json
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest", "--user-data-dir", "./browser-data"]
    }
  }

```

Agents can navigate pages, click elements, fill forms, assert content, take screenshots — all via MCP tools.

### Chrome DevTools MCP (Visual Debugging)

```json
{
  "mcpServers": {
    "chrome-devtools": {
      "command": "npx",
      "args": ["chrome-devtools-mcp@latest"]
    }
  }
```

### Agent Testing Patterns

- Use `getByRole` queries (accessibility-semantic, less brittle than `getByTestId`)
- Arrange-Act-Assert with explicit comments — agents parse this structure well
- One behavior per `it()` block
- Colocate tests: `__tests__/` or `.test.tsx` alongside the file

## 7. Auth, State Management, Data Fetching

### Auth Recommendation

| Use Case | Choice |
|---|---|
| Fastest time to working auth | **Clerk** (working in < 5 min) |
| AI agent / M2M / API key auth | **Clerk** (only one with `@clerk/agent-toolkit`) |
| Full data ownership, no vendor | **Better Auth** (successor to Lucia + Auth.js stewardship) |
| Already using Supabase | **Supabase Auth** (50k MAU free) |
| Never use for new projects | **Lucia Auth** (deprecated 2025) |

### State Management (Three Layers)

| Layer | What | Tool |
|---|---|---|
| Remote/Server state | DB/API data | RSC `async/await` + TanStack Query for live updates |
| URL state | Filters, pagination, search | **nuqs** |
| Client UI state | Sidebar open, modals, theme | **Zustand** (simple) or Jotai (atomic) |

~90% of state in a typical SaaS dashboard is remote or URL state. Only ~10% needs Zustand/Jotai. Redux is effectively dead for new Next.js projects.

### Data Fetching Patterns

| Pattern | When |
|---|---|
| Server Component `async/await` | Initial page load data |
| Server Actions (`'use server'`) | Mutations (forms, CRUD) — ~95% of server code |
| TanStack Query `useMutation` | Client-side optimistic updates, real-time polling |
| Route Handlers (`app/api/`) | Webhooks, external API consumers, custom response formats ONLY |

### ORM: Drizzle (Best for Agents)

```typescript
// Schema is TypeScript — no codegen step
import { pgTable, text, timestamp, uuid } from 'drizzle-orm/pg-core'
export const users = pgTable('users', {
  id: uuid('id').primaryKey().defaultRandom(),
  email: text('email').notNull().unique(),
})
type User = typeof users.$inferSelect  // instant type inference
```

Why Drizzle > Prisma for agents: TypeScript-native (no DSL switching), no `prisma generate` step, SQL-proximity, 7.4kb vs Prisma's 1.6MB bundle.

6. **Turborepo** — Add now for CI caching or defer until second app/package appears?
